# Отчёт о работе — превращение конфига Claude Code в самообучающийся организм

**Дата:** июнь 2026
**Репозиторий:** https://github.com/sergeeey/Claude-cod-top-2026
**Суть:** аудит + доведение разрозненного набора инструментов (агенты/скиллы/хуки/правила/память) до состояния адаптивной самообучающейся системы.

---

## 0. С чего начали

Запрос был аудитить репозиторий, а вырос в стратегическую задачу:
> «У меня много всего (агенты, скиллы, инструменты), но это разрозненно, не работает как единый организм. Хочу: адаптивную методологию под проект + чтобы система училась на ошибках.»

## 1. Хронология работы

### Этап 1 — Аудит репозитория (строгий)
- Параллельные агенты: code-quality, security, architecture.
- **Ключевой урок:** и агенты, и встречная критика **переоценили баги**. Из 4 заявленных HIGH под численной проверкой (`python -c`, реальный прогон) **не выжил ни один логический баг**.
- Подтвердились только: security в NotebookLM (опц. расширение) + косметика (type hints, 1 хрупкий тест).
- Оценка: **7.5/10**.
- **Методологический вывод:** `[VERIFIED]` агента = твой `[INFERRED]`. Off-by-one / кроссплатформенные claim проверять прогоном, а не рассуждением.

### Этап 2 — Безопасность NotebookLM
- Расширение хранило живые Google-куки (`state.json`) в плейнтексте + Chrome `--no-sandbox` + не было в `.gitignore`.
- Решение: вынести в local-only (`git rm --cached`, оставить на диске).
- Позже multi-lens вскрыл: реальный риск — **куки**, а не код; проверка показала, что куки **никогда не коммитились** → утечки не было, фикс профилактический.

### Этап 3 — Обновление личного конфига до v3.8.0
- Локальный `~/.claude` отставал на 69 коммитов от GitHub.
- Синхронизированы agents/rules/hooks/skills (+22 скилла), сохранены личные (`humanizer`, `last30days`).
- Подключены 19 новых хуков через мерж `settings.json` (с сохранением permissions/theme).
- Полный бэкап перед изменениями.

### Этап 4 — Новый скилл `multi-lens`
- Извлечён полезный кусок из «Методологии Бойко» — 20 аналитических линз.
- Создан через skill-creator, добавлен в репо.

### Этап 5 — Аудит зрелости системы (главный)
Три параллельных агента + численная верификация. **Оценка 3.7/10:**

| Ось | Было | Диагноз |
|-----|------|---------|
| Связность | 5/10 | 3 конфликта ломали пары агентов |
| Адаптивность | 4/10 | методология вручную, нет авто-детекции типа проекта |
| Самообучение | **2/10** | контур «зомби»: пишет в пустоту, не читает обратно |

### Этап 6 — Реализация (A + C)
- **A (связность + обучение):** оживлён контур памяти (`_auto/patterns.md` + `learning_log.md`), сняты 3 конфликта (покрытие 70→80%, линтер flake8/black→ruff, счёт агентов 9→13).
- **C (Диспетчер):** classifier-хук + dispatcher-скилл + патч routing-policy.
- **A1 (Рефлексия):** замкнута петля write→read — `session_start` читает накопленные уроки в начале сессии.

### Этап 7 — Дистрибуция
- Всё перенесено в репо, инсталляторы (`install.sh`/`.ps1`) сеют `_auto/` память.
- Залито в GitHub (`main`).

### Этап 8 — 4 улучшения из живой обратной связи
Пойманы при реальном использовании на проектах:
1. **Stale-copy guard** — предупреждает «открыл распакованный ZIP, а не git-клон».
2. **Git-identity check** в install — предупреждает о коммитах-заглушках.
3. **Карта различий скиллов** — skeptic/research = режимы, не дубли.
4. **Условная загрузка правил** — EstimandOps/FL не висят на простом фиксе.

## 3. Методология — как это работает

### Тип проекта → методология (Диспетчер)
| Тип | Признак (по намерению в README/CLAUDE.md) | Грузит | Пропускает |
|-----|-------------------------------------------|--------|-----------|
| research | гипотезы, фальсификация, estimand | FL Full + EstimandOps + skeptic-triggers | — |
| data-science | датасеты, метрики модели | EstimandOps + валидация на реальных данных | FL Full |
| production | деплой, библиотека, CI/CD | reviewer + tester ≥80% + FL Standard | estimand-ops, FL Full |
| mvp | прототип, PoC | скорость, FL Micro | тесты, estimand, skeptic |
| unonboarded | нет `.claude/` | онбординг (создать CLAUDE.md + activeContext) | — |

**Гибрид:** хук даёт черновик по сигналам (контент важнее структуры папок), LLM арбитрит спорные случаи.

### Цикл обучения
```
fix-коммит → pattern_extractor → patterns.md [×N]
SubagentStop → ace_reflector → playbook.md
        ↓
session_start читает топ-уроки → модель стартует, зная прошлые ошибки
```

### Принципы (правила-фундамент)
- **Evidence Policy:** каждый факт помечен `[VERIFIED]/[INFERRED]/[HYPOTHESIS]/[UNKNOWN]`.
- **Audit-verification-gate:** `[VERIFIED]` агента = твой `[INFERRED]`; HIGH/MEDIUM не доходят до вывода без tool-проверки.
- **Falsification Ladder:** claim валиден только после controls + baseline + stress-test + caveats.
- **Skeptic-triggers:** авто-скепсис на «100%», «0 failures», круглые числа.
- **Multi-lens:** 20 линз для систематического разбора (поиск слепых зон).

## 4. Как запустить на другой машине

### Windows (PowerShell)
```powershell
git clone https://github.com/sergeeey/Claude-cod-top-2026.git
cd Claude-cod-top-2026
.\install.ps1
```

### Mac / Linux (bash)
```bash
git clone https://github.com/sergeeey/Claude-cod-top-2026.git
cd Claude-cod-top-2026
./install.sh
```

Инсталлятор автоматически:
- копирует хуки (включая `project_classifier`), скиллы (включая `dispatcher`, `multi-lens`, `humanizer`), агентов, правила;
- регистрирует SessionStart-хук;
- **сеет `memory/_auto/`** (patterns.md + learning_log.md) — петля обучения работает сразу;
- предупреждает, если `git config user.email` не настроен.

### После установки (обязательные шаги)
1. `git config --global user.email "твоя-почта"` + `user.name` — чтобы коммиты были подписаны.
2. Адаптировать `~/.claude/CLAUDE.md` (секция IDENTITY) под себя.
3. Перезапустить Claude Code.
4. На каждом своём проекте: `/dispatcher` или «настройся на проект» — определит тип и методологию.

### Что НЕ переносится между машинами (по design)
- Накопленные уроки `patterns.md` (память локальна для каждой машины — это правильно).
- Персональный `CLAUDE.md` (ставится шаблон, адаптируй).

## 5. Как пользоваться на новом проекте

```
1. cd твой-проект && claude
2. SessionStart: classifier пишет project_profile.md (тип + что грузить/пропускать)
   + предупредит, если это распакованный ZIP без .git
3. /dispatcher — подтверждает тип, объявляет методологию
4. Работаешь — нужные правила/агенты подтягиваются под тип проекта
5. fix-коммит → урок копится; следующая сессия его прочитает
```

При сомнении «какой из похожих скиллов взять» → `docs/skill-disambiguation.md`.

## 6. Итог

| Ось зрелости | Старт | После работы |
|--------------|-------|--------------|
| Связность | 5/10 | конфликты сняты, карта различий |
| Адаптивность | 4/10 | Диспетчер + классификатор + условные правила |
| Самообучение | 2/10 | петля write→read замкнута |

Система из «набора органов в банках» стала **единым организмом**: понимает тип проекта, грузит ровно нужное, учится на своих ошибках между сессиями, и ставится на любую машину одной командой.

**Главный сквозной урок сессии:** не доверять заявленному (ни агентами, ни людьми) без прогона инструментом. Большинство «критических багов» испарялись под численной проверкой — а реальные дыры (мёртвый контур обучения, stale-папка) находились только при честной верификации.
