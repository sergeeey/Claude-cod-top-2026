# Estimand Bridge — дизайн заговаривает с проверкой

`estimand.md` определяет, ЧТО важно (MCID — порог значимости) и как обрабатывать события (ICE — стратегия, не imputation). Но Skeptic и PR-проверки работают по **общим** триггерам и твоих определений не знают. Результат: ты дисциплинированно спроектировал — а автоматика это не enforce-ит. Этот мост читает estimand.md и генерит проверки **под него**.

## Когда применять
Есть `estimand.md` (или `claim.md`) И нужно оценить результат/вывод. Перед тем как поверить «всё сошлось» — прогони через мост.

## Шаг 1 — Прочитать estimand

Найди и прочитай `estimand.md` (обычно `experiments/<id>/estimand.md`). Извлеки:

| Поле | Зачем мосту |
|------|-------------|
| **Question type** (descriptive/predictive/causal) | causal без DAG → авто-провал |
| **Endpoint** + units | что именно меряется |
| **Summary measure** | risk_difference / F1_delta / … (не HR/OR в гетерогенной популяции!) |
| **MCID** | порог: результат ниже = «не действовать», даже если p<0.05 |
| **ICE + strategy** | выбывшие/события обработаны стратегией, НЕ средним/LOCF |

Если estimand.md нет или MCID/ICE — плейсхолдеры → **СТОП**: «нечего enforce-ить, сначала заполни estimand».

## Шаг 2 — Сгенерировать проверки для Skeptic

Из полей собери **точечные** falsification-вопросы (не общие). Шаблон:

```markdown
## Skeptic-вопросы из этого estimand
1. [MCID] Результат = {X}. MCID = {порог}. Если |X| < MCID → результат 
   ПРАКТИЧЕСКИ НЕзначим, даже при p<0.05. Проверь: X ≥ MCID?
2. [ICE] События {ICE_name} обработаны стратегией '{strategy}' 
   или их втихую заимputировали средним/LOCF? Imputation вместо ICE = провал.
3. [summary measure] Мера — {measure}. Если это HR/OR в гетерогенной 
   популяции → noncollapsible, дрейфует. Проверь корректность.
4. [causal] Если question_type=causal → есть DAG + 4 проверки 
   identifiability? Нет → claim невалиден независимо от чисел.
5. [estimand drift] Estimand задан ДО данных? Если после — это 
   рационализация под результат.
```
Затем отдай эти вопросы скиллу `skeptic` (context asymmetry: только claim + estimand + сырые числа).

## Шаг 3 — Go/No-Go чек-лист (критерии решения)

```markdown
## Estimand Gate — pass/fail
- [ ] Результат ≥ MCID (иначе: статзначимо ≠ важно → NO-GO)
- [ ] Каждый ICE обработан стратегией, не imputation
- [ ] Summary measure не HR/OR в гетерогенной популяции
- [ ] (если causal) DAG + identifiability присутствуют
- [ ] estimand.md датирован ДО доступа к данным
- [ ] Написано "что результат НЕ значит" (≥3 пункта)

Любой [ ] не отмечен → НЕ объявлять успех. Downgrade до [NEEDS-WORK].
```

## Шаг 4 — Вердикт

```markdown
## Вердикт моста
- MCID enforce: {результат vs порог} → PASS/FAIL
- ICE: {обработка корректна?} → PASS/FAIL
- Skeptic-вопросы переданы: {да/нет}
- Итог: GO / NO-GO / NEEDS-WORK
```

## Принцип честности
Мост не заменяет Skeptic — он его **наводит** на твои собственные критерии. И не смягчает: если результат < MCID, это NO-GO, как бы красиво ни выглядел p-value. «Статистически значимо» ≠ «практически важно» — ровно для этого ты задавал MCID.

## Чем НЕ является
Не для проектов без estimand.md. Не считает статистику. Не выносит physics/causal claim — проверяет, что claim соответствует тому, что ты сам определил измерять.
