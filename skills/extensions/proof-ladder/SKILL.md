# Proof Ladder — Программа Доказательства Гипотезы

## Зачем

Большинство исследователей либо собирают доказательства хаотично, либо сразу берутся за самый дорогой эксперимент. Этот скил строит оптимальный маршрут: от теоретической правдоподобности до независимой репликации — с конкретными метриками на каждом уровне.

Ключевое отличие от обычного планирования: скил **читает твой проект** перед тем как что-то спрашивать. Если гипотеза уже в activeContext, он её найдёт. Если какой-то уровень лестницы уже пройден — он скажет откуда стартуем, а не с нуля.

---

## Шаг 0 — Автоматически читай контекст проекта

Это первое что делаешь — до любых вопросов к пользователю.

Читай последовательно, останавливаясь когда нашёл нужное. Все пути — **относительно корня текущего проекта** (cwd где запущен Claude Code):

```
1. CLAUDE.md              → домен проекта, стек, цели, ограничения
2. .claude/memory/activeContext.md  → активные гипотезы, текущие эксперименты
3. experiments/           → list директорий (что уже пробовали, какие данные есть)
4. null_results/INDEX.md  → что уже опровергнуто — КРИТИЧНО, не повторять
5. parked/INDEX.md        → что отложено и почему
6. .claude/memory/decisions.md → архитектурные решения (какой стек, какие данные)
```

**Что делать с тем что нашёл:**
- Нашёл активную гипотезу → используй её, не спрашивай "какую гипотезу?"
- Нашёл null_results для этой гипотезы → скажи сразу: *"Эта гипотеза уже опровергнута [дата]. Начинать заново?"*
- Нашёл prior experiments → покажи откуда стартуем на лестнице (не с Уровня 1)
- Нашёл данные в experiments/ → отметь какие уровни уже подкреплены
- Ничего не нашёл → задай **один вопрос**: "Какую гипотезу доказываем?"

Начни ответ блоком `[ИЗ ПРОЕКТА]` с тем что нашёл — это показывает пользователю что скил прочитал контекст, а не выдаёт шаблон.

**Если файлы не существуют** — не падай, не жалуйся. Просто пропусти и двигайся дальше.

## Шаг 1 — EstimandOps L0 Gate

Классифицируй гипотезу перед тем как строить лестницу. Тип вопроса определяет что будет считаться доказательством.

| Тип | Вопрос | Стандарт доказательства |
|-----|--------|------------------------|
| **Descriptive** | "Что такое X в популяции P?" | Репрезентативная выборка + воспроизводимость |
| **Predictive** | "Каким будет X для нового случая?" | Out-of-sample accuracy > baseline |
| **Causal** | "Что изменится если сделать A?" | RCT или квази-эксперимент + DAG |

Напиши одно предложение-эстиманд:
> *"Мы оцениваем [MEASURE] [ENDPOINT] для [POPULATION], сравнивая [INTERVENTION] vs [COMPARATOR]."*

Напиши что результат **НЕ** означает (минимум 2 пункта) — это защищает от переинтерпретации.

Для causal-гипотез: нужен DAG. Без него — downgrade до predictive или descriptive.

**Артефакт для causal:** перед запуском экспериментов создай `estimand.md` по шаблону `experiments/_template/estimand.md` (требование `estimand-ops.md`). proof-ladder строит план — но сам артефакт нужен до экспериментов.

**Если L0 уже выполнен** (результат есть в activeContext.md) — используй его, не повторяй.

---

## Шаг 2 — Быстрая Фальсификация (3 мира)

Перед тем как строить лестницу — проверь что гипотеза не очевидно ложна.

Построй 3 альтернативных объяснения тех же данных:

1. **Data World** — данные неправильно измерены или выборка смещена
2. **Method World** — есть конфаундер или артефакт метода
3. **Mechanism World** — эффект реален, но по другой причине

Для каждого: вероятность (низкая/средняя/высокая) + различающий тест.

**Вердикт:**
- Все три = низкая → продолжай к Шагу 3
- Один = высокая → скажи это явно: *"Мир N требует проверки перед инвестицией в лестницу"*
- Два+ = высокая → ПИВОТ: гипотеза нуждается в переформулировке

Детальный протокол 5 миров → `sci-evidence falsify` если нужно глубже.

## Шаг 3 — 7-Уровневая Доказательная Лестница

Для каждого уровня — статус и следующий шаг. Если уровень уже пройден (нашёл в projects/) — отметь `[DONE]` и не описывай заново.

Детальный протокол каждого уровня → `references/ladder.md`

**Быстрый обзор:**

| Уровень | Что это | Стандарт успеха |
|---------|---------|-----------------|
| **1. Теоретическая правдоподобность** | Гипотеза не противоречит известному | Нет фундаментальных противоречий |
| **2. Литературная поддержка** | ≥3 статьи косвенно поддерживают | Систематический поиск, нет опровергающих |
| **3. Слабые наблюдательные данные** | Паттерн виден в реальных данных | Эффект p<0.05, ES>0.2 |
| **4. Реплицируемый статистический сигнал** | Эффект устойчив в нескольких датасетах | ≥2 независимых источника, ES>0.3 |
| **5. Механистическое объяснение** | Есть объяснение ПОЧЕМУ это работает | Промежуточные переменные измерены |
| **6. Контролируемый эксперимент** | Эффект при вмешательстве | Pre-registered RCT или ablation |
| **7. Независимая репликация** | Другая команда воспроизвела | ≥1 независимая репликация |

## Шаг 4 — Экспериментальный Маршрут

Предложи 4 варианта эксперимента:

- **Самый дешёвый** — минимум ресурсов, быстрая проверка
- **Самый быстрый** — результат за 48-72 часа
- **Самый убедительный** — максимальная доказательная сила
- **AI/симуляция** — что можно проверить без реального мира

Для каждого: ресурсы, время, что получим, какой уровень лестницы закрывает.

## Шаг 6 — Роадмап

```
24 часа:  [1-2 конкретных действия — что можно сделать сегодня]
7 дней:   [milestone — что должно быть готово к концу недели]
30 дней:  [уровень доказательной базы к концу месяца]
6 месяцев: [где должна быть гипотеза: предрегистрация / публикация / kill]
```

**4 пути (выбери один по ситуации):**
- **Кратчайший** — если нужен результат за 30 дней (sales demo, грант, пивот-решение)
- **Надёжный** — если репутация на кону (публикация, регуляторный claim)
- **Дешёвый** — если нет лаборатории / бюджета (только биоинформатика, публичные данные)
- **Прорывной** — если потенциальный upside оправдывает высокий риск провала

## Литературный Поиск

Детальный план поиска → `references/search.md`

Быстрые правила:
- Ищи **старые термины** до современной формулировки гипотезы
- Ищи **негативные результаты** (PubMed filters: "negative results")
- Проверь **смежные домены** — доказательство может существовать под другим именем
- Проверь `null_results/INDEX.md` в проекте — не повторяй то что уже опровергнуто

---

## Запреты

- Не пиши "доказано" без независимой репликации
- Не строй Уровень 6 если Уровень 3 показывает p>0.1
- Не предлагай дорогой эксперимент если дешёвый ещё не сделан
- Не игнорируй null_results — если гипотеза уже в индексе, скажи это в первых строках
