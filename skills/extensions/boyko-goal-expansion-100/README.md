# boyko-goal-expansion-100

Глубокое исследовательское расширение цели: строит пространство решений из
до 100 различных, содержательно уникальных способов достижения цели —
не список поверхностных лозунгов.

## Установка

Часть репозитория Claude-cod-top-2026 — устанавливается стандартно вместе
со скиллами (`install.sh` → `install_extension_skills` / `sync_global_skills`,
либо просто наличие в `skills/extensions/` этого репозитория).

## Вызов

```
/boyko-goal-expansion-100 [quick|deep|frontier|audit|prototype]

Цель:
<явная формулировка цели>

Критерий успеха:
<что будет считаться успехом>

Ограничения:
<ресурсы, время, риск>

Исключить:
<что явно не предлагать>
```

Если цель уже понятна из контекста разговора/файлов проекта — можно
вызвать без явного описания, скилл извлечёт `GoalContext` сам
(см. `templates/input-template.md`).

## Режимы

| Режим | Идей | Когда использовать |
|---|---|---|
| `quick` | ≤30 | Быстрый обзор, минимальный поиск |
| `deep` (по умолчанию) | ≤100 | Полноценное исследование с источниками |
| `frontier` | ≤100 | Высокая ставка — фундаментальная научная/инженерная проблема, нужен adversarial audit |
| `audit` | 0 новых | Проверить уже созданный список 100 вариантов на дубли/ошибки |
| `prototype` | — | Топ-5 из готового списка → технические задания |

## Параметры (через GoalContext, см. `templates/input-template.md`)

`primary_goal`, `success_criteria`, `target_system`, `current_state`,
`known_constraints`, `known_attempts`, `available_resources`,
`excluded_approaches`, `evidence_standard`, `time_horizon`,
`risk_tolerance`, `requested_count` (по умолчанию 100), `output_language`.

## Пример запроса

```
Используй boyko-goal-expansion-100 в режиме frontier.

Цель:
Получить математически проверяемый способ связать локальную надёжность
ИИ-агента с глобальным риском длинного workflow.

Критерий успеха:
Хотя бы 10 идей должны иметь реализуемый недельный прототип, а 3 идеи —
чёткую математическую постановку.

Ограничения:
Один разработчик, Python, существующие логи агентов, без обучения
собственной фундаментальной модели.

Исключить:
Поверхностные аналогии и предложения вида «просто использовать больше агентов».
```

## Пример сокращённого результата

```
# 1. Формализация цели
Цель: связать локальную надёжность агента с глобальным риском workflow.
Feasibility: 6/10.

# 5. Сто вариантов (сокращено до 3 для примера)

## 1. MDP-based reliability propagation
Type: extension | Evidence: inference
Core mechanism: моделировать workflow как MDP, локальная надёжность —
вероятность корректного перехода, глобальный риск — произведение по цепочке.
Cheapest test: смоделировать toy-workflow из 5 шагов, посчитать пропагацию
ошибки аналитически за один день.
Falsifier: если реальные логи агентов показывают немарковскую зависимость
ошибок (корреляция между несмежными шагами) — модель не годится as-is.
...

## 47. [cross_domain_transfer] Эпидемиологическая модель распространения ошибок
Source domain: epidemiology (SIR-модели)
Transferred mechanism: скорость распространения дефекта как R0-подобный параметр
Target problem: ошибка в одном шаге workflow "заражает" последующие шаги
Structural correspondence: оба — процессы распространения состояния по
графу с вероятностями перехода
Cheapest test: посчитать эффективное R0 по историческим логам провалов workflow
...
```

(Полный вывод — 100 карточек, топ-12, три портфеля, план прототипирования —
см. `templates/final-report.md` для полной структуры.)

## Ограничения

- Не заменяет реальный литературный поиск специализированным скиллом,
  если тот доступен (см. `docs/living-skills.md` по поводу non-repo-tracked
  личных скиллов вроде `/lit-search`) — использует `WebSearch`/`WebFetch`
  напрямую.
- Feasibility ≤3 не отменяет генерацию идей, но меняет их состав (больше
  toy-моделей и способов опровержения, меньше «готовых решений»).
- Никогда не придумывает источники, DOI, авторов — если не уверен, что
  источник существует, помечает `<unknown>`.

## Правила работы с источниками

Первичные и авторитетные источники в приоритете; SEO-статьи — не
основание для научных выводов; проверять дату (особенно материал после
2024); не цитировать непрочитанное; при расхождении источников — показать
конфликт явно, не выбирать один молча.

## Запуск валидатора

```bash
python skills/extensions/boyko-goal-expansion-100/scripts/validate_output.py <путь-к-отчёту.md>
python skills/extensions/boyko-goal-expansion-100/scripts/detect_duplicates.py <путь-к-отчёту.md> [--threshold 0.5]
```

Оба — read-only, ничего не удаляют автоматически. `validate_output.py`
возвращает exit code 1 при критических находках (не хватает обязательных
полей, дубли ID, оценки вне диапазона). `detect_duplicates.py` только
печатает вероятные пары дублей с общими токенами — решение об объединении
принимает человек/агент вручную.

## Расширение списка исследовательских линз

Полный каталог из 25 линз — `references/domain-lenses.md`. Чтобы добавить
новую линзу — дописать в тот же файл с кратким описанием связи с типовыми
задачами; список не привязан к коду, чисто справочный markdown.
