# /build auto — Автономный режим выполнения плана

## Когда использовать

У тебя есть **утверждённый план** (список задач) и ты хочешь чтобы Claude
выполнил все шаги подряд — без паузы на подтверждение после каждого,
с коммитом после каждого среза и остановкой только при падении тестов.

## Предусловия (STOP если не выполнены)

1. План явно утверждён пользователем ("го", "да", "выполняй", "approved")
2. Ветка — НЕ main/master (проверить `git branch --show-current`)
3. `git status` — нет незакоммиченных изменений
4. Baseline тесты проходят: `pytest -x -q 2>&1 | tail -5`

## Протокол выполнения

### Шаг 0 — Объявить режим

```
🤖 BUILD AUTO — автономный режим
Задач: N | Ветка: <branch> | Стоп: тест упал / деструктивная операция
```

### Шаг 1 — Для каждой задачи из плана:

```
━━━ Задача N/M: <название> ━━━
```

1. **Реализовать** минимальный рабочий срез (только то что требует задача)
2. **Тест**: `pytest -x -q 2>&1 | tail -10`
   - Упал → **СТОП** (см. ниже)
   - Прошёл → продолжить
3. **Коммит**: `git add -p` + `git commit -m "feat(<scope>): <задача>"`
4. Перейти к следующей задаче

### Шаг 2 — Финал (после всех задач)

```
✅ BUILD AUTO COMPLETE
Выполнено: N/N задач | Коммитов: N | Тесты: зелёные
Следующий шаг: /ship или ручная проверка
```

## Правила остановки (HARD STOP)

Остановить немедленно и доложить при:

| Триггер | Действие |
|---|---|
| Тесты упали | СТОП — показать traceback, ждать инструкций |
| Деструктивная операция (DROP, rm -rf, reset --hard) | СТОП — спросить подтверждение |
| Задача требует архитектурного решения | СТОП — описать выбор, ждать |
| Scope creep (задача выросла в 2+ файлов неожиданно) | СТОП — уточнить границы |
| Уверенность < 80% в правильности реализации | СТОП — описать неопределённость |

## Правила выполнения

- **Один коммит = одна задача** — не смешивать
- **Не рефакторить** то что не входит в задачу
- **Не добавлять** фичи не из плана (scope guard)
- **ruff/mypy** после каждого Python-файла: `ruff check <file> && mypy <file>`
- Если задача мала (< 5 строк) — объединить с следующей, сообщить

## Интеграция с другими скиллами

- **Перед**: `/plan` или `/tdd-workflow` для создания плана
- **После**: `/ship` для релиза, `/code-review` для проверки
- **При падении**: `/code-debugging` для диагностики
- **Для безопасного кода**: `sec-auditor` агент после завершения
