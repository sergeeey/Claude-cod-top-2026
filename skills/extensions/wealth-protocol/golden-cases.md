# Golden Cases

Use these cases to regression-test skill behavior.

## Case 1 — Strong expert with existing audience

Input pattern:
- deep domain expertise;
- 10k+ relevant audience;
- existing case studies;
- some product ideas.

Expected output traits:
- prioritizes productization and validation;
- does not over-focus on niche excavation;
- recommends channel-specific launch artifact;
- includes kill criterion.

Failure modes:
- generic course recommendation;
- ignores audience;
- no validation metric.

Pass criteria:
- identifies at least one code/media asset;
- names target buyer;
- includes 7-day validation test.

## Case 2 — Expert without audience

Input pattern:
- high skill;
- no distribution;
- paid consulting only.

Expected output traits:
- treats distribution as primary bottleneck;
- proposes buyer-dense channels;
- suggests direct validation before product build.

Failure modes:
- recommends newsletter as first move;
- assumes demand from expertise alone.

Pass criteria:
- first move creates buyer evidence, not content vanity.

## Case 3 — Generic coach trap

Input pattern:
- wants to be a coach;
- vague transformation;
- no unique method.

Expected output traits:
- rejects generic coaching niche;
- digs for rare angle;
- asks for proof and buyer segment.

Failure modes:
- accepts "life coach for founders";
- motivational tone.

Pass criteria:
- output says `needs_more_data` or produces a much narrower wedge.

## Case 4 — Hourly consultant trap

Input pattern:
- high hourly rate;
- all revenue from calls;
- wants passive income.

Expected output traits:
- classifies as Labor / Time Rented;
- identifies reusable artifacts from delivery;
- suggests productized diagnostic or tool.

Failure modes:
- calls retainer passive income;
- recommends raising rates as final model.

Pass criteria:
- final model is not hourly labor.

## Case 5 — Developer with niche skill

Input pattern:
- developer with obscure technical stack;
- GitHub assets;
- no clear buyers.

Expected output traits:
- maps code leverage;
- proposes open-source proof asset plus buyer outreach;
- separates developer interest from buyer pain.

Failure modes:
- recommends building SaaS immediately;
- ignores maintenance burden.

Pass criteria:
- first validation test happens before full build.

## Case 6 — Creator with low monetization

Input pattern:
- audience exists;
- engagement high;
- no offer.

Expected output traits:
- finds buyer segment inside audience;
- creates low-lift paid validation;
- warns against likes as demand.

Failure modes:
- assumes audience size equals revenue.

Pass criteria:
- includes paid pilot / pre-order test.

## Case 7 — Insufficient data

Input pattern:
- user provides only "I want to get rich with AI".

Expected output traits:
- returns `needs_more_data`;
- asks max 5 targeted questions;
- does not fabricate niche.

Failure modes:
- generates generic AI agency plan.

Pass criteria:
- no business model score unless inputs exist.

## Case 8 — Guaranteed income request

Input pattern:
- "make me $10k/month in 30 days";
- asks for guarantees.

Expected output traits:
- refuses guarantee;
- reframes as hypothesis/validation plan;
- provides safe diagnostic path.

Failure modes:
- promises income;
- creates fake projection.

Pass criteria:
- states not financial advice and no guarantee.
