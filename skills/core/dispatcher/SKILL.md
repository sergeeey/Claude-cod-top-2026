# Dispatcher — адаптивная методология под проект

Одна методология на все проекты — это либо overkill (полный FL на CSS-фикс), либо дыра (research без EstimandOps). Диспетчер решает: **тип проекта × тип задачи → какую методологию загрузить.** Это «нервный узел», который связывает разрозненные правила/агенты/скиллы в адаптивный организм.

## Как работает (гибрид: хук + LLM)

1. **Читай авто-профиль.** Хук `project_classifier.py` при старте сессии пишет `.claude/memory/_auto/project_profile.md` — тип проекта + уверенность + сигналы. Прочитай его первым.
2. **Если уверенность HIGH** → доверяй вердикту хука, переходи к шагу 4.
3. **Если `ambiguous` / LOW** → арбитрируй сам (это твоя роль в гибриде): прочитай `CLAUDE.md` и `README` проекта, пойми **намерение** (что это и зачем), вынеси вердикт. Структура папок врёт (у config-тулкита есть `experiments/`, как и у research-репо) — судить по содержанию.
4. **Определи тип задачи** (через `routing-policy`): research / fix-1-2 / feature-3+ / TDD / debug / security.
5. **Применяй матрицу** (ниже) → объяви пользователю: «Проект X, задача Y → гружу [правила/агенты/скиллы]».

## Типы проектов

| Тип | Признаки намерения | Методология |
|-----|--------------------|-------------|
| **research** | проверка гипотез, научные claims, фальсификация | FL Full-Ladder + EstimandOps (L0 gate) + skeptic-triggers. Claims → `[VERIFIED]/[HYPOTHESIS]` |
| **data-science** | датасеты, метрики модели, ML-пайплайн | EstimandOps L0 + валидация на РЕАЛЬНЫХ данных (`[VERIFIED-REAL]`, не synthetic). FL Standard |
| **production** | деплой, CI/CD, библиотека, сервис | reviewer обязателен + tester ≥80% + FL Standard. security-audit перед релизом |
| **MVP** | прототип, proof-of-concept, ранняя версия | Скорость > строгость. Тесты опциональны. FL Micro (PR-inline). builder solo |
| **unonboarded** | нет `.claude/` | Запусти онбординг: спроси цель/стек → создай `CLAUDE.md` + `activeContext.md` |

## Матрица решений (проект × задача)

```
research   × hypothesis  → FL Full + EstimandOps + sci-hypothesis + skeptic
research   × quick-fix   → FL Standard (не Micro — в research даже фикс влияет на выводы)
data-sci   × experiment  → EstimandOps L0 + проверка на реальных данных + skeptic-triggers
production × feature      → reviewer + tester(≥80%) + FL Standard
production × security     → review-squad + security-audit + FL Full
production × quick-fix    → FL Micro + reviewer
MVP        × feature      → builder solo, tests optional, FL Micro
any        × debug         → hypothesis-arbiter + skeptic (конкурирующие гипотезы)
any        × refactor      → architect (Step-Back) + reviewer
```

## Формат вывода

ВСЕГДА коротко и явно, чтобы пользователь видел выбор:

```markdown
## Диспетчер
- **Проект:** <тип> (уверенность <HIGH/MEDIUM/LOW>, источник: профиль-хук / мой арбитраж по README)
- **Задача:** <тип>
- **Гружу методологию:** <конкретные правила + агенты + скиллы из матрицы>
- **НЕ гружу:** <что осознанно пропускаю и почему — напр. "EstimandOps не нужен для prod-фикса">
```

## Принцип «не больше нужного»

Главная ценность — **осознанно НЕ грузить лишнее**. Полный FL на правку опечатки = шум, который топит сигнал. Всегда явно называй, что пропускаешь и почему — это защищает от тихого расползания тяжёлой методологии на все задачи (та самая болезнь «всё всегда включено», ради лечения которой Диспетчер и создан).

## Когда профиль расходится с реальностью

Хук судит по сигналам и иногда ошибается (геномный research с `tests/`+CI может выглядеть как production). Ты — арбитр. Если по `CLAUDE.md`/`README` видно иное намерение, чем в профиле — **переопредели вердикт и обнови `project_profile.md`** одной строкой с пометкой `[LLM-override: причина]`. Это и есть гибрид: хук — быстрый черновик, ты — финальное решение.
