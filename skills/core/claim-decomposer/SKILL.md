# Claim Decomposer

## Зачем

Целостный анализ сложного утверждения — ловушка. Когда claim состоит из 5–10 шагов,
ошибка в шаге 3 маскируется корректностью шагов 1, 2, 4, 5. Системные противоречия
(атом A предполагает X, атом D предполагает не-X) вообще невидимы без полного графа.

Этот скилл делает одно: строит граф атомов, находит где граф рвётся, собирает обратно.
Всё остальное (фальсификация, consilience, evidence roadmap) — делегировать существующим скиллам.

---

## Шаг 0 — Context Lock

До разложения: зафикси что именно анализируем.

```
CLAIM: [точная формулировка — одно предложение или цитата]
SOURCE: [откуда claim — статья, код, проект, разговор]
SCOPE: [что входит в анализ, что явно исключено]
QUESTION_TYPE: descriptive / predictive / causal
```

Если causal → немедленно: нужен DAG + estimand до шага 1.
Если нет формулировки → задай один вопрос: "Что именно анализируем?"

---

## Шаг 1 — Claim Register

Выпиши все суб-утверждения из claim как нумерованный список.
Каждый атом — одно простое, фальсифицируемое утверждение.

**Формат:**
```
C1: [суб-утверждение]  [TYPE: empirical/logical/definitional/causal]  [EVIDENCE: cite/none]
C2: [суб-утверждение]  [TYPE]  [EVIDENCE]
...
```

**Правила:**
- "И" в одном предложении → разбить на два атома
- Запрещено: "возможно", "наверное", "скорее всего" — только чёткие, проверяемые утверждения
- Если атом нефальсифицируем → отметить `[UNFALSIFIABLE]` и вынести из анализа

---

## Шаг 2 — Critical Atoms

Из всего Claim Register выдели блокирующие:

```
BLOCKING_ATOMS: [C?, C?, ...]
  — атомы, провал которых убивает весь claim

LOAD_BEARING: [C?, C?, ...]
  — атомы, от которых зависят другие (D→A связи)

INDEPENDENT: [C?, C?, ...]
  — атомы, провал которых не убивает claim
```

Отметь граф зависимостей: C2 → C4 означает "C4 опирается на C2".

---

## Шаг 3 — Math-Code Trace

**Только если claim содержит формулы, код, или численные утверждения.**

Для каждого математического атома:
1. Запиши исходную формулу/код
2. Пройди шаг за шагом: каждое преобразование → что оно делает
3. Найди где теоретическая формула расходится с реализацией
4. Проверь граничные условия (edge cases, sign conventions, index-off-by-one)

```
MATH_ATOM: C?
  FORMULA: [формула]
  TRACE:
    step 1: [преобразование] → [что получаем]
    step 2: ...
  DIVERGENCE: [где теория ≠ код, если есть]
  VERDICT: CONSISTENT / DIVERGENT / NEEDS_CHECK
```

---

## Шаг 4 — Contradiction Map

Это главный шаг. Проверь каждую пару (Ci, Cj) на совместность.

**Матрица:**

|   | C1 | C2 | C3 | C4 | ... |
|---|----|----|----|----|-----|
| C1 | — | | | | |
| C2 | ? | — | | | |
| C3 | ? | ? | — | | |

Заполни каждую ячейку:
- ✅ совместимы
- ⚠️ напряжение (требуют дополнительного условия для совместности)
- ❌ противоречат (одновременно истинными быть не могут)

**Для каждого ❌:**
```
CONTRADICTION: Ci × Cj
  WHY: [объяснение конфликта]
  RESOLUTION_OPTIONS:
    a) [вариант разрешения]
    b) [вариант разрешения]
  BLOCKS_CLAIM: yes / no
```

---

## Шаг 5 — Recomposition

После атомарного анализа и Contradiction Map — собери обратно.

```
ATOMS_SURVIVED:  [список C? которые прошли проверку]
ATOMS_FAILED:    [список C? с причиной]
ATOMS_UNCERTAIN: [список C? требующих дополнительной проверки]

CONTRADICTIONS_BLOCKING: [количество + список]
CONTRADICTIONS_RESOLVABLE: [количество + список]

CLAIM_STATUS:
  HOLD    — все blocking atoms выжили, нет blocking contradictions
  WEAKEN  — claim частично верен, укажи в каком диапазоне
  PIVOT   — claim нужно переформулировать (укажи как)
  KILL    — ≥1 blocking atom провален или blocking contradiction неразрешима

RECOMPOSED_CLAIM: [что можно утверждать после анализа — честно, без лишнего]
```

---

## Шаг 6 — Gate Decision + Handoff

Вердикт + что делать дальше:

```
GATE: GO / CONDITIONAL / STOP

IF GO:
  → Следующий шаг: [конкретное действие]
  → Рекомендуемый скилл: /consilience / /proof-ladder / /sci-evidence build

IF CONDITIONAL:
  → Что нужно проверить: [список]
  → Proxy test (если реальные данные недоступны): [что можно проверить сейчас]

IF STOP:
  → Blocking reason: [один конкретный атом или противоречие]
  → Pivot direction: [куда переформулировать]
  → Сохранить в null_results: yes / no
```

## Делегирование существующим скиллам

Claim Decomposer не делает сам:

| Задача | Делегировать |
|--------|-------------|
| 5 миров где неправ | `/sci-evidence falsify` |
| 5 путей к доказательству | `/consilience` |
| 7-уровневая лестница доказательств | `/proof-ladder` |
| 9 экспертов по гипотезе | `/hypothesis-lab` |
| Конкурирующие гипотезы | `/hypothesis-arbiter` |
| Adversarial review | `/skeptic` |

Claim Decomposer делает: граф атомов → противоречия → recomposition → handoff к нужному скиллу.

---

## Когда использовать

**Используй если:**
- Claim содержит ≥4 суб-утверждений
- Claim включает формулы или код
- Несколько источников evidence (могут противоречить)
- Кто-то сказал "это правильно/неправильно" — но ты не понимаешь почему
- Перед запуском `/consilience` на сложный проект

**Не используй если:**
- Одна простая гипотеза без подчастей → `/sci-evidence falsify`
- Нужен evidence roadmap → `/proof-ladder`
- Нужна экспертная панель → `/hypothesis-lab`
